### [КР-1] Реальный пароль и слабый JWT_SECRET в файле `.env`

**Файл:** `backend/.env`, строки 12, 16
**Проблема:** Файл `.env` содержит реальный пароль `ADMIN_PASSWORD=secret1030` и псевдосекрет `JWT_SECRET=change_me_in_production_min_32_chars`. Это не случайная строка — это читабельная фраза длиной 36 символов, которая угадывается по смыслу.
**Почему критично:** Если `.env` случайно попадёт в git (например, через `git add -f` или при инициализации репозитория до `.gitignore`), пароль от админки и секрет JWT станут публичными. Строка `change_me_in_production_min_32_chars` — не секрет, это инструкция написанная открытым текстом. JWT подписанные таким секретом могут быть подделаны.

> **Простым языком:**
> **Ситуация:** Это как хранить пин-код от банковской карты на листке приклеенном к карте. Даже если карта пока у тебя — при первой же утечке всё скомпрометировано.
> **Почему важно:** Атакующий, получивший `JWT_SECRET`, может создать любой токен и войти в админку без пароля.
> **Фикс:** Сгенерировать случайный JWT_SECRET (`node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"`) и заменить значения в `.env` на реально случайные строки перед любым деплоем.

### [КР-2] CORS настроен без ограничений — `cors()` принимает любой источник

**Файл:** `backend/server.js`, строка 12
**Проблема:** Middleware `app.use(cors())` без параметров разрешает запросы с **любого** домена (`Access-Control-Allow-Origin: *`). В продакшне это означает что любой сторонний сайт может делать authenticated запросы к API от имени пользователя.
**Почему критично:** В Sprint 4 будет реализован JWT-токен в `localStorage`. При открытом CORS злоумышленник может создать фишинговую страницу, которая делает запросы к API магазина от имени залогиненного администратора.

> **Простым языком:**
> **Ситуация:** Это как оставить дверь склада открытой с табличкой «заходи кто хочешь». В dev — удобно. В продакшне — опасно.
> **Почему важно:** XSS + открытый CORS = полный доступ к данным магазина.
> **Фикс:** В Sprint 1, когда появится реальный API, добавить в `cors()` параметр `origin` со списком разрешённых доменов. Для dev — `http://localhost:5173`, для prod — домен магазина. Читать из переменной окружения `ALLOWED_ORIGINS`.

## 🟡 ВАЖНЫЕ ПРОБЛЕМЫ (исправить в Sprint 1 или раньше)

### [ВА-1] `console.log` в продакшн-коде бэкенда

**Файл:** `backend/server.js`, строки 34–36
**Проблема:** Три вызова `console.log()` в обработчике старта сервера выводят информацию о порте и окружении. В продакшн-образе эти строки попадут в логи Docker и будут раскрывать конфигурацию.
**Почему важно:** Логи в продакшне не должны содержать структурную информацию в `console.log` — это не структурированное логирование. В будущем при добавлении логгера (winston, pino) потребуется вычищать все прямые `console.*` вызовы.
**Задача для разработчика:** При переходе к Sprint 1 заменить `console.log` на структурированный логгер или обернуть в условие `if (process.env.NODE_ENV !== 'production')`.

### [ВА-2] Отсутствует `docker-compose.prod.yml` — единственный Compose файл используется для всего

**Файл:** `docker-compose.yml`
**Проблема:** В `PLAN.md` (строка 112) явно указан файл `docker-compose.prod.yml`. Он не создан. Сейчас один файл обслуживает и dev и будущий prod. Dev-конфиг (volume монтирование кода, nodemon, Vite dev-server) категорически не подходит для продакшна.
**Почему важно:** При деплое в Sprint 5 потребуется полностью другая конфигурация: multi-stage build, nginx вместо Vite, без монтирования кода, без nodemon. Если этого файла нет — деплой либо не состоится, либо на сервер уедет dev-конфиг что снижает производительность и безопасность.
**Задача для разработчика (DEVOPS):** Создать `docker-compose.prod.yml` в корне проекта согласно плану. Он должен использовать `nginx` контейнер, multi-stage build для frontend (Vite build → nginx статика) и production-режим для backend (без nodemon, без volume-монтирования кода).

## 🟢 РЕКОМЕНДАЦИИ (улучшение качества)

### [РЕ-1] Backend Dockerfile использует `npx nodemon` вместо `npm run dev`

**Файл:** `backend/Dockerfile`, строка 19
**Проблема:** CMD запускает `npx nodemon server.js` напрямую, хотя в `package.json` есть скрипт `"dev": "nodemon server.js"`. Это расхождение между тем что запускает Docker и тем что описано в `package.json`. При добавлении флагов nodemon (например `--watch`, `--ext`) их придётся дублировать в двух местах.
**Задача для разработчика:** Изменить CMD в `backend/Dockerfile` на `npm run dev`. Так единственным источником истины о команде запуска будет `package.json`.

---

### [РЕ-2] Анонимный volume для `backend/node_modules` отсутствует

**Файл:** `docker-compose.yml`, строки 19–25
**Проблема:** В backend сервисе монтируется `./backend:/app` (весь код), но нет анонимного volume для `/app/node_modules`. Это означает что `node_modules` хост-машины (если они существуют) перезапишут `node_modules` контейнера при монтировании. В frontend этот паттерн применён правильно (строка 45: `- /app/node_modules`), в backend — нет.
**Задача для разработчика (DEVOPS):** Добавить запись `- /app/node_modules` в список volumes сервиса `backend` в `docker-compose.yml`, аналогично тому как это сделано для frontend.
