## 🎯 ANTI-DRIFT ПРАВИЛА — КАК ПИСАТЬ ЗАДАЧИ ПРАВИЛЬНО

ИИ-разработчик "плывёт" если задача написана расплывчато. Эти 7 принципов предотвращают это.

### Принцип 1: "Координаты, а не описания"

```
❌ ПЛОХО:  "Исправь функцию авторизации"
✅ ХОРОШО: "Файл backend/app/services/auth_service.py,
            строка 47, функция register() —
            заменить прямое сохранение пароля на bcrypt хэширование"
```

### Принцип 2: "Конкретика вместо абстракции"

```
❌ ПЛОХО:  "Добавь валидацию данных"
✅ ХОРОШО: "В файле backend/app/schemas/user.py добавить
            field_validator для поля password:
            минимум 8 символов, одна заглавная буква"
```

### Принцип 3: "Секция ЗАПРЕТЫ обязательна"

ИИ-разработчик склонен к "улучшениям" — добавляет лишнее, меняет то что не просили.
Секция ЗАПРЕТЫ — явный стоп-лист что НЕ ДЕЛАТЬ. Без неё разработчик "поплывёт".

### Принцип 4: "Верификация = механическая проверка"

Разработчик отлично выполняет `pytest` и `tsc --noEmit`.
Если в задаче есть эти команды — он сам увидит ошибки и исправит.
Без них — отчитается "готово" с битым кодом.

### Принцип 5: "Одна задача = один коммит"

Не смешивать бэкенд и фронтенд в одной задаче если можно разделить.
Разработчик лучше выполняет 3 маленькие задачи чем 1 огромную.

### Принцип 6: "Заставь прочитать стандарты"

Секция 0 "ОБЯЗАТЕЛЬНОЕ ЧТЕНИЕ" — не опциональная.
Разработчик буквально прочитает `AGENT_DEVELOPER.md` перед написанием кода.
Без этой секции он пишет по памяти которая могла обрезаться.

### Принцип 7: "Не доверяй отчёту — проверяй код"

Разработчик может отчитаться "всё сделано ✅" при наличии ошибок.
После каждого отчёта разработчика — читай реальные файлы через `view_file`.
Только реальный код является источником истины.
